# Seijin Photo 2026

## 概要

- **用途**：成人式後の同級会向けリアルタイム写真共有アプリ
- **目的**：参加者が投稿した写真を会場全体で共有できるようにすること
- **開発規模**：個人開発
- **運用実績**：約30名規模のイベントで実運用 / 約70枚投稿
- **担当範囲**：要件整理 / 設計 / 実装 / デプロイ / 当日運用
- **技術**：Django / Python / HTML / CSS / JavaScript / AWS Lightsail / SQLite


成人式後の同級会イベント向けに制作・運用した、リアルタイム写真共有アプリです。  
参加者がスマートフォンから投稿した写真を、会場スクリーンにその場で表示できるようにしました。

イベントでは、その場で撮った写真が各自のスマホの中だけで終わってしまうことが多く、  
せっかくの空気感や思い出を会場全体で共有しづらいと感じたことが、このアプリを作ったきっかけです。

実際に約30名規模の同級会で運用し、約70枚の写真が投稿されました。  
要件整理から設計、実装、デプロイ、当日の運用まで一人で行いました。

---

## 制作背景

成人式後の同級会では、参加者同士で写真を撮る場面が多くありました。  
ただ、撮った写真をその場でみんなと共有する仕組みがなく、  
結局それぞれのスマホに残るだけになってしまうのが少しもったいないと感じていました。

そこで、参加者がスマホから簡単に写真を投稿できて、  
それを会場スクリーンにリアルタイムで流せる仕組みがあれば、  
もっと一体感のあるイベントになるのではと思い、このアプリを作りました。

作るうえでは、以下の3点を特に意識しました。

- 当日その場で迷わず使えること
- イベント中に自然に盛り上がること
- 管理側も困らずに運用できること

---

## 想定利用者

このアプリでは、利用者を大きく3つに分けて考えました。

### 参加者
写真を投稿したり、一覧を見たり、保存やいいねをする人

### 管理者
投稿内容を確認したり、必要に応じて削除する人

### 会場スクリーン
投稿された写真を自動で表示するための画面

それぞれ必要な役割が異なるため、  
同じアプリの中でも画面や機能を分けて設計しています。

---

## 主な機能

### 参加者向け
- QRコードからログイン
- 写真投稿（1枚ずつ）
- コメント入力（任意）
- 投稿一覧表示
- 写真保存
- いいね機能（1セッション1回、トグル式）

### スクリーン表示
- 投稿写真の自動表示
- 下から上に流れるスライド表示
- 写真のピックアップ表示
- 投稿がない場合のランダム表示
- 1秒ごとの自動更新（ポーリング）

### 管理者向け
- 管理者ログイン
- 投稿一覧確認
- 投稿削除
- 画像の単体ダウンロード
- コメント確認

## 作るうえで意識したこと

### 1. 当日に迷わず使えること

このアプリはイベント会場で使う前提だったため、  
参加者がその場で迷わずアクセスできることを重視しました。

そのため、URLを入力したり検索したりする必要がないように、  
QRコードからそのままアクセスできる構成にしています。

また、アクセス導線をできるだけシンプルにするため、  
QRコード経由での利用を前提にし、通常の直接アクセスは制限しました。

写真投稿も、複数枚を一度に送るより  
1枚ずつの方がわかりやすく失敗しにくいと考え、その形にしています。

コメントも必須にはせず、  
写真だけでも気軽に投稿できるようにしました。

### 2. 使う人ごとに役割を分けたこと

参加者・管理者・スクリーンでは、  
必要な機能や見せ方が大きく異なります。

参加者にはスマホで使いやすいこと、  
管理者には確認や削除がしやすいこと、  
スクリーンには会場演出として見やすいことをそれぞれ意識しました。

単に機能を増やすのではなく、  
**「誰がどこで使うか」から考えて役割を分けたこと** が、  
このアプリの設計で特に大きい部分です。

### 3. 小規模イベントに合うリアルタイム性

スクリーン表示ではリアルタイムに近い更新が必要でしたが、  
イベント規模や安定性、実装コストも考えて、  
今回は1秒ごとのポーリング方式を採用しました。

WebSocketのような方法もありますが、  
今回の用途ではそこまで複雑な構成にしなくても十分だと判断しました。

必要以上に大きな仕組みにせず、  
このイベントに合った形にすることを意識しました。

### 4. スクリーン表示の見え方の調整

スクリーン表示では、写真をランダムに流すことで、  
画面全体に動きが出るようにしています。

ただ、写真が流れ続けるだけでは、  
表示としては成立していても、見ている人にとっては変化が伝わりにくく、  
少し単調に見えてしまうと感じました。

そこで、写真を一定時間大きく表示する **「ピックアップ表示」** を取り入れ、  
画面の中で自然に目を引くポイントが生まれるようにしました。

ピックアップ表示は、新しく投稿された写真を優先しつつ、  
投稿がない時間帯でも既存の写真の中からランダムに表示されるようにしており、  
スクリーンの動きが止まらないようにしています。

これにより、新着写真が一覧の中に埋もれにくくなるだけでなく、  
投稿が少ない時間帯でも画面に変化が生まれ、  
会場全体で写真が流れ続けている雰囲気を維持できるようにしました。

ランダムに写真が流れるだけで終わらせず、  
**スクリーン全体に変化やメリハリが感じられる見え方** になるよう意識して調整しました。

### 5. 実運用を考えた負荷と管理

イベントでは「ちゃんと動くこと」が最優先になるため、  
表示負荷やトラブル対応も意識しました。

画像はサーバ側でリサイズし、  
スクリーン表示時の負荷を抑えています。

また、管理者が投稿を削除できるようにすることで、  
当日の運用で困らないようにしました。

いいね機能についても、  
セッション単位で制御することで、  
シンプルな仕様の中で必要な挙動を持たせています。

## 技術選定について

今回は **「短期間で、実際にイベントで使える形にすること」** を優先して技術を選びました。

Djangoは、投稿管理や画面の分岐、管理機能をまとめて作りやすかったため採用しました。  
SQLiteも、今回のような小規模イベント用途であれば十分だと判断しています。

インフラは AWS Lightsail を使ってデプロイし、  
実際に当日使える状態まで持っていきました。

技術的に派手なことをするより、  
その日に安定して使えることを重視しています。
