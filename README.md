# Seijin Photo 2026

成人式後の同級会イベント向けに制作した、リアルタイム写真共有アプリです。  
参加者がスマートフォンから投稿した写真を、そのまま会場スクリーンに表示できるようにしました。

イベントの場では、その場で撮った写真が各自のスマホの中だけで終わってしまうことが多く、  
せっかくの空気感や思い出を会場全体で共有しづらいと感じたことが、このアプリを作ったきっかけです。

実際に約30名規模の同級会で運用し、約70枚の写真が投稿されました。  
要件整理から設計、実装、デプロイ、当日の運用まで一人で行いました。

## 制作背景

成人式後の同級会では、参加者同士で写真を撮る場面がかなり多くありました。  
ただ、撮った写真をその場でみんなと共有する仕組みがなく、  
結局それぞれのスマホに残るだけになってしまうのが少しもったいないと感じていました。

そこで、参加者がスマホから簡単に写真を投稿できて、  
それを会場スクリーンにリアルタイムで流せる仕組みがあれば、  
もっと一体感のあるイベントになるのではと思い、このアプリを作りました。

作るうえでは、

- 当日その場で迷わず使えること
- イベント中に自然に盛り上がること
- 管理側も困らずに運用できること

を特に意識しました。

## 想定していた利用者

このアプリでは、使う人を大きく3つに分けて考えました。

- 参加者  
  写真を投稿したり、一覧を見たり、保存やいいねをする人

- 管理者  
  投稿内容を確認したり、必要に応じて削除する人

- 会場スクリーン  
  投稿された写真を自動で表示するための画面

それぞれ必要な役割が違うので、  
同じアプリの中でも画面や機能を分けて作っています。

## 主な機能

### 参加者向け
- QRコードからログイン
- 写真投稿（1枚ずつ）
- コメント入力（任意）
- 投稿一覧表示
- 写真保存
- いいね機能（1セッション1回、トグル式）

### スクリーン表示
- 投稿写真の自動表示
- 下から上に流れるスライド表示
- 新着写真のピックアップ表示
- 新規投稿がないときのランダム表示
- 1秒ごとの自動更新（ポーリング）

### 管理者向け
- 管理者ログイン
- 投稿一覧確認
- 投稿削除
- 画像の単体ダウンロード
- コメント確認

## 作るうえで意識したこと

### 1. 当日に迷わず使えること

このアプリはイベント会場で使う前提だったので、  
参加者がその場で迷わずアクセスできることをかなり大事にしました。

そのため、URLを入力したり検索したりする必要がないように、  
QRコードからそのまま入れるようにしています。

また、パスワード入力の手間も減らしたかったので、  
QRコード経由でのアクセスを前提にして、  
通常の直接アクセスは制限するようにしました。

写真投稿も、複数枚を一度に送るより  
1枚ずつの方がわかりやすく失敗しにくいと考え、その形にしています。

コメントも必須にはせず、  
写真だけでも気軽に投稿できるようにしました。

### 2. 使う人ごとに役割を分けたこと

参加者・管理者・スクリーンでは、  
必要な機能や見せ方がかなり違うので、そこを最初に整理しました。

参加者にはスマホで使いやすいこと、  
管理者には確認や削除がしやすいこと、  
スクリーンには会場演出として見やすいことをそれぞれ意識しています。

単に機能を増やすのではなく、  
「誰がどこで使うか」を考えて役割を分けたのが、このアプリの設計で一番大きい部分でした。

### 3. 小規模イベントに合うリアルタイム性

スクリーン表示ではリアルタイムに近い更新が必要でしたが、  
イベント規模や安定性、実装の重さも考えて、  
今回は1秒ごとのポーリング方式を採用しました。

WebSocketのような方法もありますが、  
今回の用途ではそこまで複雑な構成にしなくても十分だと判断しました。

必要以上に大きな仕組みにせず、  
このイベントに合った形にすることを意識しました。

### 4. スクリーンが止まって見えないこと

イベント中にスクリーン表示が止まって見えると、  
空気が少し途切れてしまう感じがあったので、  
新規投稿がない時間でも何かしら動きがあるようにしました。

新着があればそれを優先して表示し、  
投稿が止まっている時間帯はランダム表示に切り替えるようにしています。

写真をただ並べるだけではなく、  
会場演出として自然に見えることを意識して調整しました。

### 5. 実際に使うことを考えた負荷と管理

イベントでは「ちゃんと動くこと」が最優先になるので、  
表示負荷やトラブル対応も意識しました。

画像はサーバ側でリサイズして、  
スクリーン表示時の負荷を抑えています。

また、管理者が投稿を削除できるようにして、  
当日の運用で困らないようにしました。

いいね機能についても、  
セッション単位で制御することで、  
シンプルな仕様の中で必要な挙動を持たせています。

## 技術構成

- フロントエンド：HTML / CSS / JavaScript
- バックエンド：Python / Django
- インフラ：AWS Lightsail
- データベース：SQLite
- HTTPS対応済み

## 技術選定について

今回は「短期間で、実際にイベントで使える形にすること」を優先して技術を選びました。

Djangoは、投稿管理や画面の分岐、管理機能をまとめて作りやすかったので採用しました。  
SQLiteも、今回のような小規模イベント用途であれば十分だと判断しています。

インフラは AWS Lightsail を使ってデプロイし、  
実際に当日使える状態まで持っていきました。

技術的に派手なことをするより、  
その日に安定して使えることを重視しています。

## デモ・資料

ポートフォリオ確認用として、デモ環境に再デプロイしています。

用意しているものは以下です。

- ポートフォリオ動画
- 参加者画面のスクリーンショット
- 管理者画面のスクリーンショット
- 構成資料 / 説明資料
- デモアクセス用QRコード

※ スマートフォンでのアクセスを推奨しています  
※ デモ環境のため、投稿データは変更・リセットされる場合があります

## 苦労した点

特に苦労したのは、  
スマホでの使いやすさと、スクリーンでの見栄えを両立することでした。

具体的には、

- 縦横比の違う画像を崩さず表示すること
- スマホとスクリーンの両方に合うUIにすること
- 投稿一覧、ピックアップ、自動更新を自然に組み合わせること
- 実運用を想定した細かい調整

あたりにかなり時間を使いました。
